# 2.1 - Definir piramide de testing por modulo

## Objetivo

Definir estandar de testing por modulo y por capa para todo el ciclo post-MVP.

## Razon Tecnica

Sin estrategia uniforme, cada modulo evoluciona con criterios distintos y aumenta deuda de calidad.

## Comandos Exactos

```bash
mkdir -p docs/iteracion-02-hardening-auth-calidad-y-testing/fase-2-calidad-y-testing-base/evidencia
printf "# Test Strategy\n\n- domain: unit\n- application: integration\n- presentation: contract\n- e2e: critical flows\n" > docs/iteracion-02-hardening-auth-calidad-y-testing/fase-2-calidad-y-testing-base/evidencia/test-strategy.md
```

## Ejemplo De Codigo (Si Aplica)

```txt
auth:
- domain unit tests
- application integration tests
- api contract tests
```

## Verificacion

1. Existe documento de estrategia de pruebas por capa.
2. Cada modulo MVP tiene nivel de prueba minimo definido.
3. La estrategia se referencia desde README de iteracion.

## Resultado Esperado

Base comun para implementar y evaluar tests en siguientes iteraciones.

## Criterio De Cierre

- [ ] objetivo cumplido
- [ ] verificacion ejecutada
- [ ] evidencia registrada
- [ ] documentacion relacionada actualizada

## Troubleshooting Breve (Si Aplica)

- conflicto de alcance por modulo -> priorizar flujos criticos de negocio y seguridad
- expectativas de cobertura irreales -> subir umbrales en etapas
