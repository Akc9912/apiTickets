# 6.2 - Ejecutar pruebas de contrato API

## Objetivo

Ejecutar validacion de contrato API y comparar contra baseline para detectar cambios intencionales y no intencionales.

## Razon Tecnica

El cambio de arquitectura no debe romper comportamiento sin evidencia. Esta tarea aporta trazabilidad de contrato entre fase 0 y fase 6.

## Comandos Exactos

```bash
mkdir -p docs/iteracion-01-migracion-stack-y-arquitectura/fase-6-limites-modulares-y-pruebas/evidencia
./mvnw test -Dspring.profiles.active=test
curl -s http://localhost:8080/v3/api-docs > docs/iteracion-01-migracion-stack-y-arquitectura/fase-6-limites-modulares-y-pruebas/evidencia/openapi-post-migration.json
diff -u docs/iteracion-01-migracion-stack-y-arquitectura/fase-0-estabilizacion-tecnica-minima/evidencia/openapi-baseline.json docs/iteracion-01-migracion-stack-y-arquitectura/fase-6-limites-modulares-y-pruebas/evidencia/openapi-post-migration.json > docs/iteracion-01-migracion-stack-y-arquitectura/fase-6-limites-modulares-y-pruebas/evidencia/openapi-diff.patch || true
```

## Ejemplo De Codigo (Si Aplica)

```txt
openapi-diff.patch
- /api/legacy/endpoint
+ /api/account/v1/profile
```

## Verificacion

1. Tests de contrato ejecutan sin fallas criticas.
2. Se genera openapi-post-migration.json.
3. Existe diff documentado contra baseline de fase 0.

## Resultado Esperado

Cambios de contrato API quedan controlados, justificados y rastreables.

## Criterio De Cierre

- [ ] objetivo cumplido
- [ ] verificacion ejecutada
- [ ] evidencia registrada
- [ ] documentacion relacionada actualizada

## Troubleshooting Breve (Si Aplica)

- diff enorme y ruidoso -> separar cambios esperados por modulo y volver a comparar
- tests inestables -> aislar casos dependientes de datos no deterministas
