# 1.3 - Establecer estrategia de mapeo DTO-domain-entity

## Objetivo

Definir estrategia unica de mapeo entre DTOs de entrada/salida, modelos de dominio y entidades de persistencia.

## Razon Tecnica

Sin estrategia de mapeo clara se filtran detalles de persistencia al dominio y aumenta deuda tecnica. Engineering exige mappers explicitos.

## Comandos Exactos

```bash
mkdir -p docs/iteracion-01-migracion-stack-y-arquitectura/fase-1-fundacion-arquitectura-por-capas/evidencia
mkdir -p src/main/java/com/poo/miapi/module/auth/infrastructure/mapper
mkdir -p src/main/java/com/poo/miapi/module/account/infrastructure/mapper
mkdir -p src/main/java/com/poo/miapi/module/ticket/infrastructure/mapper
printf "# Mapping Guidelines\n\n- DTO <-> Application Command\n- Domain <-> Persistence Entity\n- No exponer Entity en controller\n" > docs/iteracion-01-migracion-stack-y-arquitectura/fase-1-fundacion-arquitectura-por-capas/evidencia/mapping-guidelines.md
```

## Ejemplo De Codigo (Si Aplica)

```txt
public Ticket toDomain(TicketEntity entity) { ... }
public TicketEntity toEntity(Ticket domain) { ... }
```

## Verificacion

1. Cada modulo migrado tiene carpeta de mapper en infrastructure.
2. Existe guia de mapeo documentada en evidencia.
3. No se retorna Entity directamente desde capa presentation.

## Resultado Esperado

Mapeos predecibles, con responsabilidad explicitamente ubicada y baja fuga entre capas.
