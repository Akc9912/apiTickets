# 5.4 - Verificar flujo funcional de ticket

## Objetivo

Verificar end-to-end el flujo principal de ticket con arquitectura desacoplada.

## Razon Tecnica

Refactor sin validacion funcional puede introducir regresiones silenciosas. Esta tarea prueba comportamiento observable del modulo.

## Comandos Exactos

```bash
TOKEN_SUPPORT="<jwt_supabase_support>"
TOKEN_ADMIN="<jwt_supabase_admin>"
curl -i -X POST http://localhost:8080/api/tickets/v1/create -H "Authorization: Bearer $TOKEN_SUPPORT" -H "Content-Type: application/json" -d '{"title":"Ticket prueba","description":"Falla funcional"}'
curl -i -H "Authorization: Bearer $TOKEN_ADMIN" http://localhost:8080/api/tickets/v1/all
```

## Ejemplo De Codigo (Si Aplica)

```txt
POST /api/tickets/v1/create
GET  /api/tickets/v1/all
```

## Verificacion

1. Se puede crear ticket con rol autorizado.
2. Listado de tickets funciona para rol admin.
3. Errores de autorizacion responden con codigo HTTP correcto.

## Resultado Esperado

Flujo operativo de ticket confirmado despues del desacople arquitectonico.

## Troubleshooting Breve (Si Aplica)
