# 1.1 - Definir catalogo de eventos por modulo

## Objetivo

Definir catalogo de eventos de dominio e integracion para auth, account, ticket y product con contratos claros.

## Razon Tecnica

Sin un catalogo explicito de eventos, cada modulo publica mensajes ad hoc y se pierde compatibilidad entre consumidores.

## Comandos Exactos

```bash
mkdir -p docs/iteracion-05-eventos-notificaciones-y-observabilidad/fase-1-backbone-eventos-modulares/evidencia
rg "Event|DomainEvent|publish" src/main/java/com/poo/miapi/module -g "*.java"
./mvnw -DskipTests compile
```

## Ejemplo De Codigo (Si Aplica)

```txt
Eventos iniciales sugeridos:
- AccountVerifiedEvent
- TicketCreatedEvent
- TicketPriorityChangedEvent
- ProductMessageConvertedEvent
```

## Verificacion

1. Cada modulo tiene lista de eventos propietarios y consumidores.
2. Cada evento define payload minimo y version de contrato.
3. No hay dependencia circular entre modulos por eventos.

## Resultado Esperado

Catalogo de eventos versionado y utilizable para desacople progresivo del monolito modular.

- [ ] objetivo cumplido

## Troubleshooting Breve (Si Aplica)

- eventos duplicados con distinto nombre -> unificar nomenclatura por bounded context
- payload excesivo -> reducir a datos minimos para consumidores
